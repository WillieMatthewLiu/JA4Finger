## Context

当前仓库只有一个空白项目骨架，需要从零开始定义 JA4 指纹分析工具的首版结构。需求要求工具既能以 CLI 方式分析 PCAP 文件，也能持续监听指定网卡流量并由外部进程管理器后台托管，并优先交付可试用、可验证的 JA4 能力，同时保留 TCP 指纹扩展能力。

这意味着首版设计必须同时处理两类输入源、一类明确的 JA4 协议边界和一类未来扩展点。如果 live 模式与 pcap 模式分别独立实现，后续演进会很快出现重复逻辑与不一致行为。因此需要先在设计层面统一采集、解析、指纹和输出这四段流水线，并把首版指纹范围收敛到能够清晰命名和验证的 TLS JA4。

## Goals / Non-Goals

**Goals:**
- 交付一个基于 Go 的单程序工具，支持 `live` 与 `pcap` 两种运行模式。
- 复用同一条分析流水线处理实时抓包和离线文件分析。
- 首版实现基于 TLS ClientHello 的 JA4 指纹识别，并输出源地址、源端口、协议类型、指纹类型和指纹值。
- 在指纹引擎层预留 TCP 指纹扩展接口，避免未来新增能力时重构主流程。
- 将错误处理限定在单报文或单会话级别，避免个别异常中断整体分析。
- 提供可重复验证的样例输入与稳定输出约束，使首版达到“可试用”而非仅“能运行”。

**Non-Goals:**
- 首版不追求极限性能优化，例如零拷贝、内核旁路或多队列调优。
- 首版不引入复杂的守护进程管理、服务注册或远程配置中心能力。
- 首版不承诺实现完整的 TCP 指纹规则集，只定义扩展接口与挂载点。
- 首版不提供数据库、消息队列或远程 API 输出，只提供本地日志输出。
- 首版不实现 HTTP 指纹或 JA4H。
- 首版不对无法提取完整 TLS ClientHello 的流量做强行补全，缺失关键握手数据时直接跳过。

## Decisions

### 1. 使用 Go 实现单二进制双模式工具

工具将以一个 Go 二进制提供 `live` 和 `pcap` 两个子命令，而不是拆成两个独立程序。

- 推荐原因：代码复用最高，部署和运维最简单，适合首版快速落地。
- 备选方案：
  - Rust 实现：性能上限更高，但当前首版瓶颈更可能在抓包与解析链路，开发复杂度不划算。
  - 两个独立程序：短期可行，但会增加共享逻辑分叉与行为漂移风险。

### 2. 采用统一分析流水线

系统将拆分为 `capture`、`decoder`、`fingerprint`、`output`、`engine` 五个模块。`live` 模式与 `pcap` 模式只负责提供不同输入源，之后统一进入同一条分析路径。

- 推荐原因：能保证两种运行模式在协议解析、指纹计算和错误处理上行为一致。
- 备选方案：
  - 为 `live` 和 `pcap` 各自实现独立逻辑：实现速度看似更快，但后续维护成本更高。

### 3. 首版只支持 TLS JA4，并将 TCP 能力抽象为可插拔引擎

`fingerprint` 层将定义统一接口，首版只挂载 TLS JA4 实现，未来可以继续接入 TCP 指纹或单独的 HTTP/JA4H 实现。

- 推荐原因：既满足“JA4 首版可试用”的目标，又避免把 HTTP/JA4H 规范不清的问题混入当前版本。
- 备选方案：
  - 把 TCP 先写成占位逻辑：会引入虚假的完成感，不如明确只保留接口。
  - 同时实现 HTTP/HTTPS：会让“JA4”命名与规范边界失焦，增加验收不确定性。

### 4. 输出采用结构化日志，后台运行依赖外部进程管理

首版统一输出到标准输出，字段至少包括 `src_ip`、`src_port`、`protocol`、`fingerprint_type`、`fingerprint`。后台运行能力通过 `live` 模式持续执行实现，是否以守护进程方式托管交由 systemd、supervisor 或用户环境处理。

- 推荐原因：避免把守护进程管理与指纹识别耦合在一起，便于测试与部署。
- 备选方案：
  - 内建 daemonize 行为：会引入平台差异与调试复杂度。
  - 直接写文件：运维灵活性不如 stdout/stderr 管道。

### 5. 错误处理采用“单事件失败不影响全局”的容错策略

协议解析失败、缺失关键字段或个别报文损坏时，系统记录错误并继续处理后续流量。只有启动参数错误、PCAP 文件无法打开、网卡无法访问等初始化错误才导致命令失败退出。

- 推荐原因：这类网络分析工具必须在脏流量条件下保持可用。
- 备选方案：
  - 遇到任意解析错误立即退出：会导致 live 模式不可用，且不符合运维预期。

## Risks / Trade-offs

- [抓包库与平台权限差异] → 通过在设计中明确 live 模式依赖宿主机抓包权限，并在实现阶段优先选择成熟 Go 抓包库来降低风险。
- [JA4 规则边界与实现细节不清] → 先将首版目标定义为“TLS JA4 可试用并可验证”，在 spec 中把输入、输出和行为约束写清楚，避免把算法细节写成不可交付的承诺。
- [实时模式与离线模式出现行为偏差] → 通过统一 `engine` 和共享 `fingerprint` 实现控制分叉。
- [日志输出过于简单导致后续集成成本上升] → 先固定最小字段集合，并保留未来扩展 JSON 字段的空间。
- [TLS 握手数据不完整导致无法产出 JA4] → 在规范中明确这类流量应记录失败并跳过，而不是输出不可信的指纹。

## Migration Plan

1. 在仓库中初始化 Go 项目结构并定义 CLI 入口。
2. 实现输入源抽象，使 `live` 和 `pcap` 两种模式都能输出统一数据包事件。
3. 实现 TLS 解析与 JA4 指纹逻辑。
4. 接入结构化日志输出并完成错误处理。
5. 通过样例 PCAP 和 live 抓包验证行为，确认首版满足基本需求后再进入后续增强。

该变更是新能力引入，不涉及历史版本迁移。若实现失败，回滚方式为移除新增代码与命令入口，不影响现有业务。

## Open Questions

- live 模式是否默认增加 BPF 过滤策略以降低非 TLS 流量噪声。
