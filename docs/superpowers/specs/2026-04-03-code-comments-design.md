# 代码中文注释设计

## 背景

当前仓库源码主要位于 `src/` 与 `tests/`，总代码量接近 4000 行，但现有中文注释几乎为空白。用户希望为所有代码补充中文注解，同时将测试文件的注释控制在关键说明层面，避免测试代码被大量逐行解释淹没。

本设计只解决“补充中文注释”这一件事，不改变任何业务逻辑、数据结构、命令行为或测试语义。

## 目标

- 为仓库中的每个函数补充中文说明。
- 为公开接口、命令参数和关键类型补充适合 Rust 生态的中文文档注释。
- 为复杂逻辑增加少量必要的中文行内注释，帮助阅读但不制造注释噪音。
- 对测试文件仅补充关键场景说明和辅助函数用途说明，不做逐断言解释。

## 非目标

- 不重构模块结构。
- 不改动锁文件、依赖、构建脚本或发布产物。
- 不把简单语句逐行翻译成中文。
- 不修改 `target/`、`dist/` 等构建输出目录中的文件。

## 范围

本次注释补充范围限定为：

- `src/*.rs`
- `src/commands/*.rs`
- `tests/*.rs`

当前已确认的目标文件包括：

- `src/lib.rs`
- `src/main.rs`
- `src/cli.rs`
- `src/aggregator.rs`
- `src/capture.rs`
- `src/config.rs`
- `src/fingerprint.rs`
- `src/output.rs`
- `src/pipeline.rs`
- `src/runtime.rs`
- `src/commands/mod.rs`
- `src/commands/aggregate.rs`
- `src/commands/daemon.rs`
- `src/commands/pcap.rs`
- `tests/cli.rs`

## 方案选择

本次采用“分层注释”方案，而不是逐行翻译式注释。

原因如下：

- 用户要求覆盖所有函数，但没有要求把每一行语句都解释一遍。
- 当前仓库逻辑性较强，尤其是 `fingerprint`、`pipeline`、`output`、CLI 测试等模块；逐行注释会严重拉高维护成本。
- Rust 代码更适合在函数、类型、字段和关键逻辑边界处补充解释，而不是对语法细节做重复说明。

## 注释覆盖规则

### 公开接口

对公开结构体、枚举、公开函数和需要在 CLI 层暴露语义的字段，优先使用中文 `///` 文档注释。注释重点包括：

- 该项的职责
- 关键输入或字段含义
- 输出或副作用
- 特殊约束或上下文

### 私有函数

所有私有函数在函数定义前补充中文说明，优先使用简短的函数头注释，说明：

- 该函数负责什么
- 输入如何被解释
- 返回值或失败条件

如果函数逻辑非常简单，则保持一行说明；如果函数涉及格式转换、筛选、解析或边界处理，则使用 1 到 2 行说明。

### 函数体内部

只在以下情形补充中文行内注释：

- 复杂协议解析或格式转换
- 容易误读的边界条件
- 需要解释提前退出或去重原因的分支
- 测试数据构造中不直观的协议字段段落

不会为显而易见的赋值、调用或标准控制流逐行加注释。

### 测试文件

测试文件只保留关键说明，主要包括：

- 辅助函数负责创建什么测试夹具
- 一组测试大致覆盖什么命令或行为
- 某个关键测试在保护什么回归场景

不会对每个断言、每个临时路径工具函数或每段字节数组都做逐条解释。

## 按文件落地策略

### 第一批：入口与调度层

目标文件：

- `src/lib.rs`
- `src/main.rs`
- `src/cli.rs`
- `src/commands/mod.rs`
- `src/commands/aggregate.rs`
- `src/commands/daemon.rs`
- `src/commands/pcap.rs`

这一批优先明确：

- 程序入口如何调度各命令
- CLI 子命令和参数的职责
- command runner 的输入输出语义

### 第二批：业务与运行时模块

目标文件：

- `src/aggregator.rs`
- `src/capture.rs`
- `src/config.rs`
- `src/output.rs`
- `src/pipeline.rs`
- `src/runtime.rs`

这一批重点补充：

- 数据结构与辅助函数职责
- 事件、日志、配置、流水线处理之间的关系
- 较复杂的筛选、聚合、渲染与状态处理逻辑

### 第三批：高复杂度逻辑与测试

目标文件：

- `src/fingerprint.rs`
- `tests/cli.rs`

这一批采用更克制的注释策略：

- 函数级说明必须完整
- 行内注释只放在协议解析、指纹计算和复杂测试场景构造的关键处
- 避免让超长文件因注释过密而降低可读性

## 注释风格约束

- 默认使用中文交流性描述，代码标识、命令名、协议名、API 名称保持英文。
- 优先解释“为什么存在”和“做了什么”，而不是解释 Rust 基础语法。
- 注释尽量短句，避免大段堆砌。
- 与现有少量英文文档注释保持语义一致，但改为中文表达。
- 如果局部逻辑已经非常清晰，则不为满足“数量”而添加低价值注释。

## 风险与控制

### 风险

- 注释改动文件多，补丁体积会明显扩大。
- 复杂模块若注释过密，反而会削弱代码本身的可读性。
- 文档注释与现有字段/函数语义若不精确，可能引入误导。

### 控制措施

- 坚持按批次补注释，避免无序扫改。
- 对简单函数只写短说明，不强行堆叠注释。
- 对复杂逻辑先识别分段边界，再决定是否添加行内注释。
- 完成后统一执行格式和测试验证，确保注释改动未破坏代码状态。

## 验证计划

本次任务不改业务逻辑，但在声称完成前仍执行与仓库匹配的验证：

- `cargo fmt --check`
- `cargo test`

如果其中任一命令失败，需要明确区分：

- 是否由本次注释改动引入
- 是否为仓库原有问题
- 当前仍存留的风险

## 实施完成标准

满足以下条件时，认为本次任务完成：

- 范围内所有函数均有中文注释说明。
- 公开接口与关键类型具备合适的中文文档注释。
- `tests/cli.rs` 仅补充关键说明，没有出现大规模逐断言注释。
- 注释风格在不同文件间保持基本一致。
- `cargo fmt --check` 与 `cargo test` 通过，或失败原因被清晰披露。
